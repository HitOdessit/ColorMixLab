# Retrospective: Building ColorMixLab Without Typing

This is the human-facing retrospective on building ColorMixLab end-to-end with AI tooling. For pure project context (architecture, build commands, conventions), see [CLAUDE.md](CLAUDE.md). For deeper technical decisions, see [ARCHITECTURE.md](ARCHITECTURE.md) and [docs/adr/](docs/adr/).

## The premise

ColorMixLab was built from scratch using [Cursor](https://cursor.com) and [Claude Code](https://docs.anthropic.com/en/docs/claude-code). Every line of Kotlin, Swift, Gradle configuration, GitHub Actions YAML, and Markdown — including this document — was generated by AI under direct human prompting and review. **No code was hand-written.**

The goal was not to prove AI can write a `Hello World`. The goal was to find out whether a non-trivial, cross-platform mobile application — KMP, Jetpack Compose, SwiftUI, CI, tests, release pipeline — is buildable end-to-end this way today, and where the seams are.

## What made this interesting

- **Cross-platform from day one.** The project started as a single-platform Android color-mixing game and evolved into a Kotlin Multiplatform app with a native iOS port. The AI handled the KMP migration: `expect`/`actual` platform abstractions, shared game logic, two native UI layers staying in sync.

- **The iOS port was the hardest part.** Bridging Kotlin `StateFlow` to SwiftUI cleanly requires either [SKIE](https://skie.touchlab.co/) or [KMP-NativeCoroutines](https://github.com/rickclephas/KMP-NativeCoroutines). Both add Gradle plugin and compiler-plugin complexity for what is, at the end of the day, a turn-based color-mixing game. The pragmatic call: a `Timer.publish(every: 0.1)` polling `gameController.gameState.value` and republishing via `@Published`. 100ms is well below the perception threshold for this app's interactions, and the trade-off is documented inline and in [ADR-0003](docs/adr/0003-ios-stateflow-polling-bridge.md).

- **Math challenge distractors needed iteration.** The first cut at "wrong answers" for multiplication questions was random integers in a plausible range — easy for a kid to eliminate. Getting to pedagogically sound distractors (off-by-one factor errors, squared factors, near-misses, nearby multiples) took multiple prompt cycles. This is where AI got close on the first try and needed deliberate human direction to nail the last 20%.

- **Animation performance was a recurring fight.** The game completion celebration — a multi-phase 50+ particle explosion over 10 seconds — dropped frames in the first implementation. Each particle had its own `animateFloatAsState`, and the recomposition graph couldn't keep up on mid-tier devices. The current implementation uses a single master `Animatable` clock and computes positions inside `Canvas { }`'s `DrawScope` from pre-allocated `FloatArray` buffers. Zero per-frame allocations, zero recomposition. Documented in [ADR-0004](docs/adr/0004-drawscope-particle-rendering.md).

- **The cleanup pass was AI-driven too.** Removing 54 development markdown files, deleting dead code, switching `_gameState.value = ...` reads to `_gameState.update { }` for atomicity, modernizing the build (Gradle version catalog, SDK bumps) — all done through Claude Code with a deliberate human review pass after.

## What the AI got right on the first try

- MVVM architecture with `GameController` as the single source of truth
- The color-mixing math (RGB averaging, Euclidean similarity)
- Tier-based color unlocking with random selection from candidate pools
- Time-windowed leaderboard queries (today / week / month / all-time)
- Haptic feedback integration on both platforms via the `expect`/`actual` pattern

## What the AI got wrong (and how the human caught it)

- **Non-atomic state mutations.** Initial implementations used `_gameState.value = _gameState.value.copy(...)` everywhere. Looks fine; it's a non-atomic read-modify-write. The timer-tick coroutine and the UI thread can both mutate concurrently and silently lose updates. Required a deliberate sweep to convert every mutation site to `update { }`. The AI didn't flag this as a problem until prompted — once prompted, it found and fixed every instance correctly. See [ADR-0002](docs/adr/0002-atomic-stateflow-updates.md).

- **Dead code accumulation.** Across iterations the AI left behind helper functions, debug logs, and stale `// TODO` comments that nothing called. Cleanup needed a focused human-prompted pass.

- **Animation regressions across iterations.** Each new animation feature sometimes silently broke a previously-working one. Snapshot tests would have caught this; we didn't have them at the time. (See [ROADMAP.md](ROADMAP.md) — Paparazzi now planned.)

- **Over-eager documentation.** The AI generated 54 markdown files documenting every change as the work happened. Useful in the moment, clutter for a public repo. They're curated under [docs/dev-notes/](docs/dev-notes/) — a small representative sample kept, the rest archived out.

## Lessons learned

1. **AI excels at cross-platform consistency.** Having one AI generate both the Android and iOS implementations meant game logic, scoring, and UX stayed in sync without manual coordination. This is a real, repeatable benefit and probably the strongest case for AI-assisted KMP work.

2. **Code review is still essential.** The AI produces working code quickly. It also produces code with subtle thread-safety issues, dead branches, and over-engineering. A human review pass that *specifically* looks for those is non-negotiable.

3. **Prompts are the source of truth, not the code.** The conversation that produced the code matters more than the diff, and is harder to capture. Future projects should commit prompt transcripts alongside the code.

4. **Constrain the problem before generating.** "Build a color-mixing game" produces worse output than "create `MathQuestionGenerator` that produces multiplication questions with the following 6 distractor strategies, each documented…". The narrower and more specific the prompt, the better the result.

5. **AI is great at boilerplate; humans are still better at scope.** Deciding *what not to build* — staying out of online multiplayer, IAP, or analytics — was a human call. The AI happily would have built those.

## Reproducing this

[docs/reproduce.md](docs/reproduce.md) contains a curated set of example prompts that would rebuild major parts of this project from scratch. They are not the exact original prompts — those are conversational, long, and personal — but they are a reasonable starting point for anyone who wants to see whether AI-assisted mobile development is real or hype.

The honest answer: it is real, with caveats above.
